# CLAUDE.md

Guidance for future Claude Code sessions working in this repository.

## Project Overview

Mutiso.AI is a construction-site management app for the Kenyan market (KES currency, `en-KE` locale formatting, references to OSHA 2007, WIBA 2007, DOSHS, NCA, KPLC, Energy Act 2019). It serves three roles: **Super Admin** (platform-wide), **Contractor/Admin** (owns sites, approves subscriptions, assigns foremen), and **Foreman** (runs one assigned site day-to-day: attendance, material deliveries/usage, site diary, photos, inventory). Foremen onboard via emailed invite links rather than self-registering.

**This is a clean-slate rebuild of an earlier project (`mutiso-main`, kept on disk as a sibling folder for reference/rollback only — do not import code from it).** The old repo accumulated schema drift (live tables with no migration record), two incompatible generations of the data model, and at least one confirmed bug (site diary entries silently missing their `site_id`). Rather than keep reconciling that repo, this workspace starts fresh: new git repo, new Supabase project, frontend rebuilt from scratch using the old app only as a UX/feature reference. Being built in staged phases — see the approved plan for the full roadmap. As of 2026-07-08, **all 7 planned stages are complete and verified** end-to-end: scaffold; core schema/auth/multi-tenant skeleton; foreman daily operations + offline-first (attendance, materials, diary, photos); Tier 1 safety/compliance (incident register, toolbox talks, inspection checklists, permit-to-work); Tier 2 quality/progress (defect/snag list, sequence-gated milestone sign-off, an aggregated site report/history feed, plus an "at a glance" stats row on the foreman dashboard); Tier 3 financial/contract administration (payroll computed from attendance, delivery-note waybill upgrade with ordered-vs-received mismatch flagging, subcontractor registry with work orders, variation orders with an approval thread); Tier 4 asset/equipment/access (tool & heavy-plant lifecycle with checkout/return history and a maintenance log, visitor sign-in/sign-out log, worker/equipment certification tracker with an expiring-soon badge); and Tier 5 environmental (NEMA-oriented waste disposal log with quantity/disposal-method/disposal-partner tracking, plus dust/noise/spill incident reporting reusing Stage 3's incident register rather than a new table). Every stage was verified with real browser walkthroughs and explicit RLS/constraint/gate checks via direct API calls, not just build/lint. Also added ahead of plan, at the user's request: a minimal Super Admin site-approval screen and a Sign In tab (the original plan only specced sign-up). **The originally-planned roadmap is now fully built** — future work is user-directed extensions/refinements rather than remaining plan stages.

## Architecture & Conventions (carried forward from the old repo, still the target pattern)

- `src/pages/` — route-level components, one per route, default-exported
- `src/components/` — shared components
- `src/components/forms/` — modal-style forms, one per action
- `src/components/ui/` — shadcn-generated primitives; don't hand-roll new patterns here, add via the CLI
- `src/hooks/` — one file per domain, `.tsx` extension by convention even without JSX
- `src/integrations/supabase/` — `client.ts` + generated `types.ts` (regenerate via `supabase gen types typescript --project-id kmcgcqnuxixsxqwigfir` once Stage 1's schema exists — do not hand-maintain this file)
- `src/lib/` — `utils.ts` (`cn()` helper), `offlineQueue.ts` (IndexedDB queue primitives), plus static content files as needed

**Conventions to keep from the old app:**
- Form components as full-screen overlays (`fixed inset-0 z-50 bg-background/95 backdrop-blur-sm animate-fade-in`), `siteId` + `onClose` props, mutation with toast on both success and error, `queryClient.invalidateQueries` on success.
- TanStack Query key convention: `['resource-name', ...ids]`.
- Custom shadcn `Button` variants beyond the stock set: `construction`, `action`; custom sizes `xl`, `touch` — see `src/components/ui/button.tsx`.
- Kenya-specific locale formatting inline: `toLocaleTimeString('en-KE', {...})`.
- Role-based view selection at the page level (`Index`-equivalent), not the router level.

**Conventions being deliberately changed from the old app (don't regress to the old pattern):**
- Attendance and any "is this person/item present" matching must use a foreign key, never a name-string comparison (the old `AttendanceForm` matched on `worker.full_name` string equality — two same-named workers would have collided).
- Every table that logs a site-scoped event must have `site_id NOT NULL` at the schema level, not just as an application-level convention — the old repo's `ActivityForm` silently dropped `site_id` because nothing enforced its presence.
- Forms use react-hook-form + zod, not raw `useState` + manual checks.
- Loading states use skeletons, not bare spinners.

## Database

**Live schema (36 migrations in `supabase/migrations/`):**
- Stage 1: `profiles`, `user_roles` (`app_role` enum: `super_admin`/`contractor`/`admin`/`foreman`, all four from day one — the old repo had this drift undocumented), `sites`, `site_assignments` (partial unique index enforces one active assignment per foreman at the DB level, plus a symmetric `site_assignments_one_active_per_site` index added 2026-07-13 — one active foreman per site), `invites` (as of 2026-07-13, invite creation requires an email and `consume_invite()` verifies the claiming user's auth email matches it, and auto-voids other pending invites for the same site once one is claimed), `notifications` (`recipient_id` XOR `recipient_role` via a check constraint — direct notifications vs. role-broadcast ones).
- Stage 2: `workers_master` (per-site roster), `attendance_log` (FK to `workers_master.id`, `unique(site_id, worker_id, date)` — replaces the old repo's name-string matching entirely, both the match logic and the double-mark prevention are now DB-enforced), `site_diary_log` (`site_id NOT NULL`), `materials_delivered` + `material_inventory` + `material_usage_log`, `site_photos` + a private `site-photos` Storage bucket (object paths are `{site_id}/{filename}`, storage policies parse `site_id` out via `storage.foldername(name)`).
- Stage 3: `incident_log` (medium/high severity auto-notifies the site owner via an `AFTER INSERT` trigger), `toolbox_talk_log` + `toolbox_talk_attendance` (submitted together via the `create_toolbox_talk()` RPC — see below), `inspection_template` (shared reference table, seeded with 4 default checklists: PPE/Scaffolding/Electrical/Housekeeping) + `inspection_log` (results stored as jsonb, `flagged_count` computed client-side at submit time), `work_permit` (the one table in this schema where INSERT and UPDATE are governed by *different* policies on purpose — see "Permit approval gate" below).
- Stage 4: `defect_log` (a real CHECK constraint, `defect_verifier_not_fixer`, blocks the same user from both fixing and verifying — not just a UI convention), `site_milestone` (a fixed 5-stage sequence — Foundation/Structure/Roofing/Finishes/Handover — auto-seeded for every new site via an `AFTER INSERT` trigger on `sites`, and backfilled once for sites that pre-date the migration; see "Milestone sequence gate" below). No new table for the site report/history feed — `useSiteReport` just aggregates existing tables client-side for a date range.
- Stage 5: `payroll_run` + `payroll_line` (weekly runs generated from attendance via `generate_payroll_run()`, see below; split-policy RLS — SELECT open to owner+foreman, ALL owner-only, same shape as the permit gate), `materials_delivered` upgraded with `receipt_photo_url` (optional, uploaded to the same `site-photos` bucket as everything else) — originally also had `waybill_number`/`ordered_quantity` with a client-side ordered-vs-received mismatch flag, but that was removed 2026-07-14 at the user's request (dropped columns, simplified `log_material_delivery()` back down to a 7-arg signature) since it added friction without real value in practice; only received `quantity` is tracked now, `subcontractor` + `subcontractor_work_order` (standard blanket owns_site/is_assigned_foreman RLS, no split policy needed since either role can manage the registry), `variation_order` + `variation_order_response` (split-policy RLS again: INSERT scoped to `raised_by = auth.uid()` + site access, UPDATE — the approve/reject decision — scoped to `owns_site()` only, identical reasoning to the permit gate).
- Stage 6: `tool_inventory` (covers both hand tools and heavy plant via `category`) + `tool_checkout_log` (append-only transition history) + `equipment_maintenance_log` (service/repair/inspection entries, shown only for `category = 'plant'` tools in the UI), `visitor_log` (simple sign-in/sign-out, no RPC needed — see below), `certification` (worker or equipment certs in one table, a CHECK constraint enforces `subject_type` matches exactly one of `worker_id`/`tool_id` being set).
- Stage 7: `waste_log` only (`waste_type`/`disposal_method`/`quantity`/`unit`/`disposal_partner`/optional evidence photo, standard blanket RLS, no RPC — single-table insert like diary/attendance). No new incident table — see below.
- 2026-07-08 (post-plan, user-directed): a schema gap analysis against a full construction-lifecycle data model (Project/Contract, Planning/WBS, Procurement, Field Ops, Financial, Quality, Documents, Closeout) found this schema already covers Safety/HSE, Quality, and Punch List thoroughly, but had no real WBS/BOQ, no Budget/Actual-Costs/Payment-Certificates layer, and no unified document registry. Two of the target model's cross-cutting assumptions were rejected as not fitting this codebase: no `tenant_id`/`organization_id` column (the existing `sites.owner_id` + `site_assignments` + `owns_site()`/`is_assigned_foreman()` pattern stays), and no new `app_role` values (financial/planning tables gate on the same 4 roles as everything else, not a `project_manager`/`finance`/`client` split). First slice built: `activity` (real WBS — self-referencing `parent_id` hierarchy, `planned_start/end`, `actual_start/end`, `percent_complete` with a `0–100` CHECK, `status`) + `activity_dependency` (predecessor links as a real FK join table, not a `uuid[]` column — an array can't be per-element FK-constrained) + `boq_item` (`total_amount` is a Postgres `generated always as (quantity * unit_rate) stored` column, not a hand-maintained one, so it can't drift) + `resource_plan` (planned labor/equipment/material vs. `workers_master`/`tool_inventory`'s actuals).
- 2026-07-08, second slice (user-directed): the Financial Tracking layer identified as a gap above, plus a **schedule baseline** feature the user specifically asked for ("save the original schedule of works... checks against the activities recorded... how ahead or behind"). Added `site_contract` (1:1 per site), `budget_line`, `actual_cost` (non-labor only — labor's actual cost is `payroll_line.gross_amount`, not duplicated here, same "don't build a table that already exists in another form" reasoning as `incident_log`/`waste_log`), `payment_certificate` + `generate_payment_certificate()` RPC (auto-numbers the certificate, pulls the default retention % from `site_contract`, sums every prior certificate's `net_amount_due` for `previous_payments_total` — verified directly: two sequential certs at 10% retention on KES 1M then KES 2M cumulative work correctly produced `net_amount_due` of 900,000 then 900,000 again, accounting for the first payment). `schedule_baseline` + `schedule_baseline_activity` + `save_schedule_baseline()` RPC snapshot every current `activity` row's planned dates into an immutable copy — baselines are re-saveable with history kept (most recent `locked_at` = "the current baseline"), and a later edit to the live `activity.planned_end` is verified to never change an already-saved snapshot. **`activity`'s RLS was tightened from Stage 8's blanket owner-or-foreman policy to a split shape**: only the site owner can INSERT/DELETE (WBS structure is a planning decision), and a new `BEFORE UPDATE` trigger (`enforce_activity_structural_lock`, same pattern as `enforce_milestone_sequence`) rejects a non-owner update touching `name`/`activity_code`/`parent_id`/`planned_start`/`planned_end`/`responsible_party` — a foreman can still freely update `percent_complete`/`actual_start`/`actual_end`/`status` as field truth. Verified directly: a foreman changing `planned_start` on an activity is rejected by the trigger while the same foreman changing `percent_complete` on the same row succeeds.
- **Progress/schedule variance is computed client-side** (`useScheduleProgress.tsx`), not stored — same "compute, don't store" approach as `useSiteReport`/`AtAGlanceStats`. Classifies each activity `ahead`/`on_track`/`behind` by comparing its baseline planned dates against live `actual_end` (if finished) or an expected-percent-complete-by-today calculation (if still open), with a simple "most-delayed activity" headline stat. **Deliberately not a true critical-path (CPM) calculation** even though `activity_dependency` exists to support one later — flagged as a real v1 simplification, not an oversight.
- **Contractor-facing UI was consolidated into one entry point**: `ProjectOverviewView` (`src/components/forms/ProjectOverviewView.tsx`), opened via a single "Overview" button that replaced the separate Payroll/Variations buttons on `ContractorView`'s site cards (and reachable from `ForemanDashboard`'s Quality & Progress tiles too, in a role-adapted form — the same component just conditionally hides contractor-only controls via `isContractor`, since RLS is what actually enforces the boundary either way). Sections: Progress & Schedule (activity list + baseline save), Milestones (embeds the existing `MilestonesView` as a nested overlay, zero backend change needed — it already had no role-gating), Finance Summary (rolls up contract/budget/actuals via a new `useFinanceSummary` hook, links out to `ContractView`/`BudgetView`/`PaymentCertificatesView` and the existing `PayrollView`/`VariationOrdersView`), Site History (embeds the existing `SiteReportView`).
- 2026-07-08, third slice: bulk **schedule-of-works CSV upload** (`ScheduleUploadDialog.tsx`), so a contractor doesn't have to hand-enter every activity — replaces the site's whole WBS in one atomic transaction via `replace_site_activities(p_site_id, p_activities jsonb)` (delete-then-bulk-insert has to be one RPC, not two client round-trips, or a network drop mid-operation could leave a site with zero activities). **Considered `xlsx` (SheetJS) for true Excel parsing and rejected it** — the npm-published package has two unpatched high-severity advisories (prototype pollution, ReDoS) with no fix available, and this is exactly the wrong place to accept that risk since it parses attacker-reachable user-uploaded file content. Shipped CSV-only via `papaparse` instead (no known high-severity vulns); the UI tells users to export Excel/MS Project/Primavera schedules to CSV first. Column matching is alias-based and case/punctuation-insensitive (`FIELD_ALIASES` in `ScheduleUploadDialog.tsx`) since different tools export different header names for the same concept. Date parsing accepts ISO (`YYYY-MM-DD`) as-is or assumes `DD/MM/YYYY` otherwise (this market's convention) before falling back to the browser's date parser — verified directly with a real CSV containing both formats plus a blank-name row (correctly skipped) and a blank date (left `null` rather than guessed). **Before running `npm install` for a file-parsing/data-import library, check `npm audit` before wiring it up, not after** — this one would have shipped a real vulnerability if not caught.
- 2026-07-08, fourth slice: **real WBS hierarchy for the schedule upload**, prompted by a user testing this with an actual MS Project export (nested phases like "PROJECT INCEPTION" containing "Site visit"/"site analysis"). `replace_site_activities()` now does a second pass after the bulk insert: a single set-based `UPDATE` links each row's `parent_id` by stripping the last dot-segment off its `activity_code` (the parent of `"2.2.1"` is `"2.2"`) — this is exactly what MS Project's "Outline Number" export field already encodes, so the fix was entirely about *using* data the importer already captured, not adding a new column. A code with no dot, no code at all, or a dotted code whose prefix doesn't match any uploaded row all safely degrade to top-level rather than erroring — verified directly with a deliberate gap in the numbering (a `"1.10.2"` row with no `"1.10"` parent present). `ScheduleUploadDialog`'s preview now indents rows by dot-count and flags any code that won't find a real parent match *before* upload, and `ProjectOverviewView`'s activity list now renders via a new `buildActivityTree()` helper (`useActivities.tsx`) — depth-first order with a `depth` field for indentation — instead of a flat `activity_code` sort. Verified end-to-end in the browser with a CSV mirroring the real Egret Lodge file: the post-upload tree rendered with the same nesting as the original Gantt chart.
- 2026-07-08, fifth slice: **decluttered the Overview's Progress & Schedule card** into a summary + drill-down, and added actual charts (frontend-only, no new migration). The full activity list and "Add activity" form moved out of the compact Overview card into a new full-screen `ScheduleDetailView` (opened via a "View all activities (N)" button) — the Overview card itself now shows only a `ScheduleSummaryChart` (an SVG progress ring for overall % complete + a stacked ahead/on-track/behind bar, both reusing the same success/muted/destructive tokens the classification chips already use, so the chart and the badges read as one system) plus the most-delayed callout and the Upload/Save-baseline quick actions (kept on the main card per user request, not moved). `ScheduleDetailView` also gained a `ScheduleGanttChart` (`src/components/ScheduleGanttChart.tsx`) — a real timeline: month gridlines, a sticky label column indented by `buildActivityTree()`'s depth, bars positioned by day-offset and colored by schedule-progress classification, a "today" marker line, and a hover tooltip. Verified live against a CSV mirroring the user's actual MS Project export (nested phases, nine-month range) — the rendered tree and bar positions matched the source Gantt chart.
- 2026-07-08, sixth slice: **linked the foreman's field data to the WBS, and the WBS to milestones** — closing a real gap the user pointed out: progress % and milestone status were entirely manual, disconnected from anything a foreman actually logs day to day. `site_diary_log.activity_id` (nullable, `on delete set null`) lets a foreman optionally tag a diary entry to the activity it's evidence for, via a new picker on `DiaryForm` (only rendered when the site has activities); `ScheduleDetailView` shows each activity's linked diary feed via one grouped query (`useDiaryEntriesByActivity` — fetches every diary row with a non-null `activity_id` for the site once and groups client-side, not a per-activity query, since a real uploaded schedule can easily be 50+ activities). **Deliberately not automatic**: percent_complete stays a manually-entered number — the diary link gives it supporting evidence, it doesn't compute the number.
  `activity.milestone_id` (nullable, `on delete set null`) lets the contractor tag each activity with which of the 5 fixed milestones it belongs to — joined the same owner-only guard list in `enforce_activity_structural_lock()` as `name`/`activity_code`/`parent_id`/`planned_start`/`planned_end`/`responsible_party`, since assigning an activity to a phase is a planning decision. `MilestonesView` computes readiness client-side (linked-activity count vs. how many are at 100%) and shows a **"Ready to sign off" nudge — not an automatic status write**. This was a deliberate call: milestone completion auto-populates `signed_off_at`/`inspected_by` via the existing `enforce_milestone_sequence` trigger (a real compliance record), so it should never happen silently just because every linked activity crossed 100% — a human still has to click Sign Off. **Any future "should X auto-complete based on Y" idea in this schema should default to a client-side readiness nudge, not a DB-level auto-write, whenever the target status change has audit/compliance weight** (mirrors why payroll/permits/variations are owner-gated decisions, not automatic either).

A "Full history" link was also added directly under the stats row, alongside the existing Quality & Progress tile - two entry points to the same `SiteReportView`, deliberately. (3) `SiteReportView` gained a `Today` preset and a real custom start/end date range (previously only `Last 7 days`/`Last 30 days`).
- 2026-07-08, eighth slice (frontend-only): `SiteReportView` (the shared History board used by both the foreman's "Site Report" tile and the contractor's Overview "Site History" section — one component, two entry points, per an earlier decision in this log) now groups entries by type (People Present, Materials Delivered, Materials Used, Site Activities, Incidents, Toolbox Talks, Inspections, Defects, Photos) instead of one flat interleaved list. Groups are collapsible (`TYPE_ORDER`/`TYPE_LABEL` in `SiteReportView.tsx`), show a count badge, and only render if they have entries in the selected range - the underlying `useSiteReport` hook/query is unchanged, this is purely a client-side reorganization of already-fetched data.
- 2026-07-08, ninth slice (frontend-only): `SiteReportView`'s category groups got a real categorical color system and switched from inline expand to per-category popovers, collapsed by default. **Tool checkout/return (`checkout_tool()`/`return_tool()`)**: same atomic-check-then-write shape as `log_material_delivery`/`log_material_usage` — `checkout_tool` rejects if the tool's current `status` isn't `'available'` (raises with the current status in the message), `return_tool` rejects if it isn't `'checked_out'`, and finds the open `tool_checkout_log` row (`returned_at is null`) to close rather than requiring the client to track a log id. Both RPCs do their own explicit `owns_site()`/`is_assigned_foreman()` check inside the function body — this isn't redundant with RLS, it's necessary: a SECURITY DEFINER function bypasses RLS internally, so without that explicit check any authenticated user could check out/return *any* site's tool by id. Verified directly: a second, unrelated contractor calling `checkout_tool()` against the first contractor's tool id is rejected with "Not authorized for this site" even though the RPC itself can see the row. **Any future SECURITY DEFINER RPC that takes a target row id as a parameter needs this same explicit ownership check inside the function — being SECURITY DEFINER is exactly why RLS alone won't protect it.**

**Visitor sign-in/out doesn't use an RPC, unlike tool checkout** — it's genuinely single-table (insert for sign-in, update for sign-out), so there's no atomicity gap to close. Sign-in is offline-queued like attendance; sign-out deliberately is not — see "Referential availability" below.

**Payroll computation (`generate_payroll_run(p_site_id, p_week_start, p_week_end)`)**: owner-only (raises if called by a foreman — same `owns_site()` gate as the milestone/permit patterns), computes `days_present` per worker by counting `attendance_log` rows in the date range joined to `workers_master`, then `gross_amount = days_present * daily_rate`. Verified directly: seeded 3 days of attendance at `daily_rate = 1500` for one worker, ran the RPC as both foreman (rejected: "Only the site owner can generate a payroll run") and contractor (succeeded, produced exactly one `payroll_line` with `days_present = 3`, `gross_amount = 4500`, `net_amount = 4500` before advances/deductions). Marking a line `paid` is a separate owner-only UPDATE (blanket-denied to foremen via RLS, confirmed via zero-row-return check).

**Financial authority ≠ field capture — this distinction now runs through the whole offline-queue eligibility decision.** Attendance/deliveries/usage/diary are foreman field actions and are offline-queueable (Stage 2's `submitOrQueue`). Payroll generation, marking a payroll line paid, and variation-order approve/reject are **not** offline-queued even though they're small JSON payloads — they're owner-only decisions gated by RLS that the client can't locally fake past (a foreman's device queuing a "mark paid" locally would just fail loudly on replay once reconnected, so there's no UX benefit and it adds a confusing false-positive "queued" state for an action that was always going to be rejected). Only extend the offline queue to an action if *every* authorized user of that form could plausibly be offline in the field — not just some of them.

**Referential availability is a second, distinct reason not to offline-queue something (Stage 6).** Visitor sign-*in* is queued (a plain insert, same shape as attendance); sign-*out* deliberately is not, even though any foreman doing sign-in could equally plausibly do sign-out in the field. The reason isn't an authorization gate this time — it's that sign-out is an UPDATE against a specific existing row's id, and a visitor signed in while offline has no server row (and no real id) to update until their sign-in has already synced. `useSiteVisitors` only ever lists rows that made it to the server, so the UI can only offer "sign out" for a row that's safe to update directly — there's no case where a user would even see the option for an unsynced visitor. **Before queuing any UPDATE/RPC-against-an-id, ask whether the id it targets could itself still be sitting unsynced in the same queue — if so, either don't queue it, or (like `checkout_tool`/`return_tool`) key the RPC off something resolvable server-side (the tool's current open checkout row) instead of a client-supplied log id.**

**Permit approval gate (`work_permit`)**: this is the one table where a blanket "owner or assigned foreman can do everything" policy would be wrong — a foreman must be able to request a permit but never approve their own. INSERT is scoped to `requested_by = auth.uid()` (either role); UPDATE (the approve/reject transition) is scoped to `owns_site()` only, deliberately excluding `is_assigned_foreman()`. Verified directly: a foreman calling `.update({status:'approved'})` on their own pending permit is denied by RLS (returns zero rows, not an error — see the UPDATE-silently-returns-nothing lesson below), confirmed still `pending` afterward via a service-role check. **Any future "X requests, Y approves" feature should follow this same split-policy pattern, not the blanket owner-or-foreman policy used everywhere else.**

**Milestone sequence gate (`site_milestone`)**: unlike the permit gate (an RLS/policy-level restriction), this one is enforced by a `BEFORE UPDATE` trigger (`enforce_milestone_sequence`) that rejects moving a milestone to `in_progress`/`completed` unless the previous `sequence` for that site is already `completed` — verified directly: attempting to start "Structure" (sequence 2) while "Foundation" (sequence 1) is still pending raises a Postgres exception and leaves the row unchanged, confirmed via a service-role check afterward. The same trigger also auto-populates `signed_off_at`/`inspected_by` the moment a milestone transitions to `completed`, so the client never sets those fields directly. **Any future "can't skip ahead" ordering requirement should use this same BEFORE UPDATE trigger pattern, not a client-side disabled-button check** (a disabled button is a UX nicety; the trigger is what actually prevents it, regardless of client).

**RLS pattern for every new policy**: `USING ((select auth.uid()) = owner_column)`, scoped `TO authenticated` explicitly, plus `has_role()` (SECURITY DEFINER) for admin-bypass rather than inlining a subquery. Every table gets RLS from the migration that creates it.

**Golden rule for this workspace**: no table exists on the remote project without a corresponding local migration file. If you ever find yourself about to hand-create a table via the dashboard SQL editor "just to test something," write the migration first instead.

**`owns_site()`/`is_assigned_foreman()` now also require `sites.status = 'active'`** (`20260730090800_gate_access_by_site_status.sql`), not just ownership/assignment — closing a real gap where a not-yet-approved (or later-cancelled) site's owner or foreman could still hit the API directly for anything gated by these two helpers (payroll, materials, diary, permits, milestones, tools, etc. — nearly the whole schema), even though the UI only ever showed management actions for `active` sites. This was deliberate and verified live (create a pending site → invite creation 403s → approve the site → the same invite call succeeds). Two things to know before touching either function again: (1) `sites` itself does **not** route through `owns_site()` for its own policies (it checks `auth.uid() = owner_id` directly), so a contractor can still see/manage their own pending or cancelled site rows — only *other* tables lose access; (2) `invites` was the one table that never routed through `owns_site()` at all (it only checked `invited_by = auth.uid()`, no real ownership or status check) — fixed in the same migration by adding an `owns_site()` check to its policy. **Any new table gated by "site owner" or "assigned foreman" should use `owns_site()`/`is_assigned_foreman()`, never a hand-rolled ownership check** — that's exactly the gap that let `invites` slip through unenforced for this long.

**Two hard-won RLS lessons from building Stage 1 — read before writing any new cross-table policy:**

1. **Circular RLS references cause "infinite recursion detected in policy for relation X."** The `sites` policy for "assigned foremen can view their site" queried `site_assignments`, and the `site_assignments` policy for "site owners can manage assignments" queried `sites` right back — each policy's subquery re-triggers the other's RLS evaluation, forever. Fix: wrap either side's cross-table check in a `SECURITY DEFINER` function (see `is_assigned_foreman()`/`owns_site()` in `20260706174500_fix_sites_assignments_rls_recursion.sql`) — same pattern as `has_role()`, which is why that pattern exists in the first place. **Whenever a new policy on table A needs to check table B, and a policy on B might check A back, use a SECURITY DEFINER function, don't inline the subquery.**
2. **An UPDATE that RLS filters out returns no error and silently affects zero rows** — `!error` is not proof anything happened; always check the returned row count (`.select()` + check `data.length`, or check `data` isn't null on a `.single()`). This one cost real debugging time: an invite-consumption UPDATE looked like it succeeded (no error) but had silently updated nothing, because the acting user (freshly authenticated, no longer `anon`) had no SELECT-visibility into that row under any policy — RLS-invisible rows can't be UPDATEd even with a seemingly-matching UPDATE policy's own USING clause.

**Invite consumption is a single atomic RPC (`consume_invite(p_token uuid)`, SECURITY DEFINER), not three separate client-side table writes.** The three-write version (client directly UPDATEs `invites`, INSERTs `user_roles`, INSERTs `site_assignments`) needed RLS policies narrow enough to allow "the invited user, only this row" — that shape of policy turned out to be impossible to write correctly at the table level: any policy permissive enough for the legitimate case was also permissive enough to let any authenticated user browse or claim *any* live invite platform-wide (confirmed — a second contractor could see a different contractor's invites under an earlier version of this policy set). The RPC validates the token itself internally (bypassing RLS as SECURITY DEFINER) and does the whole claim in one transaction. **Don't reintroduce direct client-side writes to `invites`/`user_roles`/`site_assignments` for the join flow — call the RPC.**

## Offline-first architecture (Stage 2)

`src/lib/offlineQueue.ts` (IndexedDB via `idb-keyval`) + `src/hooks/useOfflineQueue.tsx` (`OfflineQueueProvider`, mounted once in `App.tsx`) is the whole mechanism. Forms that need to work with spotty connectivity (attendance, deliveries, usage, diary — **not** photo upload, a multi-MB file isn't reasonable to hold in IndexedDB the way a small JSON payload is; foreman just retries photos when back online) call `submitOrQueue(op)` instead of writing to Supabase directly. Online-and-successful behaves like a normal mutation; offline, or a network failure while online, queues the operation and replays it on reconnect. A queued operation that fails on replay for a *real* reason (e.g. insufficient stock) moves to a separate "failed" list instead of blocking everything queued after it forever — only genuine network failures stay in the retry queue.

**Two real bugs surfaced only by actually testing this with the browser toggled offline — both are the kind of thing that look fine in code review and fail silently in production:**

1. **TanStack Query's `useMutation` defaults to `networkMode: 'online'`, which pauses the mutation and never calls `mutationFn` at all while `navigator.onLine` is false.** This pre-empts `submitOrQueue`'s own offline handling before it ever runs — confirmed via debug logging that the mutation didn't even start until *after* reconnecting, at which point it just fired directly rather than ever touching the local queue. **Every `useMutation` that calls `submitOrQueue` must set `networkMode: 'always'`** — see `useMarkPresent` (`useAttendance.tsx`), `useAddDiaryEntry` (`useDiary.tsx`), `useLogDelivery`/`useLogUsage` (`useMaterials.tsx`) for the pattern.
2. **`useOfflineQueue()` must be consumed via the `OfflineQueueProvider` context, never called as a bare hook.** It used to be a plain hook — every call site got its own independent queue processor with its own `'online'` event listener. Multiple components using it simultaneously (e.g. the dashboard's status banner *and* a form's mutation hook) meant reconnecting fired multiple concurrent `flush()` calls, all racing to process the same queued item — confirmed via a real 409 conflict when two flushes both tried to insert the same attendance row. Now there's exactly one `OfflineQueueProvider` instance (mounted in `App.tsx`, inside `AuthProvider`), and every `useOfflineQueue()` call reads from that single shared instance via context.

**Auth config lives in `supabase/config.toml` and needs `supabase config push` separately from `supabase db push`** (the latter only pushes migrations/schema). Two things already fixed here that a fresh `supabase init` would default wrong for this project: `[auth.email] enable_confirmations` is `false` (so `signUp()` returns an active session immediately — the whole app's UX assumes this, e.g. `JoinPage`/`Auth` navigate straight to `/` after signup) and `site_url`/`additional_redirect_urls` point at port `8080` (the app's actual dev port, not the `3000` default). If `supabase config push` errors on a Storage `databasePoolMode` schema mismatch, that's a separate, currently-unresolved CLI/Storage-service quirk — unrelated to Auth, safe to ignore until Stage 2 needs Storage buckets.

## Known Gotchas & Constraints

- **Certification/insurance "expiring soon" badges are computed client-side, not via a notifications-table trigger** (`isExpiringSoon()` in `src/hooks/useCertifications.tsx`, same 30-day-lookahead helper originally written for Stage 5's subcontractor insurance badge). The plan's Stage 6 wording said "surfaced via the notifications pattern," but every existing use of `notifications` is event-triggered (an `AFTER INSERT` firing off something that just happened — a new incident, a new permit request), and there's no equivalent "event" for a date silently getting closer; doing this properly would need a scheduled job (`pg_cron` + a periodic function), which is disproportionate for a lookahead badge and not yet set up anywhere in this project. If real push/email reminders for expiring certs become a requirement later, that's the point to add `pg_cron`, not before.
- **`CREATE OR REPLACE FUNCTION` does not replace a function when you add new parameters.** Postgres identifies a function by name **and** argument-type signature, so adding params (even with defaults) creates a second overload rather than replacing the original — confirmed while upgrading `log_material_delivery()` from 6 args to 9 in Stage 5. Fix: `DROP FUNCTION IF EXISTS public.fn(old, arg, types);` before `CREATE FUNCTION` whenever a migration changes an RPC's parameter list, not just its body.
- **A `type="date"` HTML input that's part of an optional zod field submits `""`, not `undefined`, when left blank** — react-hook-form's `register()` reads uncontrolled inputs straight from the DOM on submit, and an untouched date input's DOM value is the empty string, not `null`/`undefined`. Postgres then rejects it with `invalid input syntax for type date: ""` (a 400, not caught by the zod schema since `z.string().optional()` happily accepts `""`). Found live in Stage 5's `AddSubcontractorForm` (`insurance_expiry` field) via an actual browser submit with the date left blank — not something code review caught. Fixed at the mutation layer (`useAddSubcontractor`, `src/hooks/useSubcontractors.tsx`): coerce `insurance_expiry || null` before the insert. **Any new optional date field needs the same `value || null` coercion in its mutation hook** — don't rely on the zod schema alone to catch this.
- **The same blank-input problem hits `z.coerce.number().optional()` too, but the fix is different and the bug is nastier.** A blank number input's DOM value is also `""`, and `z.coerce.number()` coerces `""` to `0` *before* `.optional()` ever runs — so the field never ends up `undefined`, it ends up a real, valid `0`. That's worse than the date case: `0` is frequently a legitimate value (e.g. deliberately entering 0% retention), so a `value || null` coercion at the mutation layer can't tell "left blank" apart from "explicitly zero." Found live in `PaymentCertificatesView`'s "Retention %" field (defaults to the contract's rate when left blank) — leaving it blank silently sent `0` and overrode the real contract retention rate instead of falling back to it. Fix has to happen *before* coercion: `z.preprocess((v) => (v === '' || v === undefined ? undefined : v), z.coerce.number().optional())`. **Any optional numeric field where `0` is a meaningful value needs this preprocess pattern, not a post-hoc `|| null`.**
- **Vite 8 (the current default) is incompatible with `@vitejs/plugin-react-swc@3.11.0`** (peer dependency only goes up to Vite 7) — this is why Vite is pinned to `^5.4.19` rather than left at whatever `npm create vite@latest` defaults to. If you ever bump Vite, check `@vitejs/plugin-react-swc`'s peer range first.
- **`npm create vite@latest -- --template react-swc-ts` no longer works** — newer `create-vite` versions dropped the swc-specific template names. Use `--template react-ts` and then manually swap `@vitejs/plugin-react` → `@vitejs/plugin-react-swc` in `package.json`/`vite.config.ts`.
- **This machine had a Webshare proxy configured globally in `~/.npmrc` and `git config --global`** (with `strict-ssl=false` on the npm side) that was removed during this rebuild after confirming with the user it was intentional-but-no-longer-needed/blocking. If proxy-related 407 errors resurface, check `.npmrc`/`git config --global --get http.proxy` before assuming it's a network issue — a backup of the old `.npmrc` exists at `C:\Users\Administrator\.npmrc.bak_proxy` on this machine.
- **PWA icons are placeholder-only right now** (`public/manifest.json` points at `favicon.svg`) — the old repo's manifest referenced `icon-192.png`/`icon-512.png` that didn't actually exist in its `public/` folder either. Add real app icons before treating the PWA manifest as complete (relevant once offline-first work starts in Stage 2).
- **2 known vulnerabilities from `npm audit`** as of this writing, coming from pinning Vite to 5.4.x to match the old stack rather than using latest. Re-run `npm audit` periodically and weigh whether the old-stack-parity assumption is still worth it once Stage 1+ is underway.
- No test framework exists yet — see Tech Stack note above.
- **Email domain validation on the public `signUp()` endpoint is stricter than the admin API.** Testing signup flows with a made-up domain (e.g. `@test.mutiso.ai`) gets rejected by `supabase.auth.signUp()` even though `admin.auth.admin.createUser()` accepts it fine — use `example.com` or a real-looking domain when testing the public signup form manually, or use the admin API for test fixtures that don't need to exercise the actual public form.
- **A growing number of leftover test-user accounts exist in the live Supabase project** from Stage 1–4 verification (emails containing `test.mutiso.ai`, `rls.`, or `s2contractor.`/`s2foreman.`/`browsertest.`/`foreman.`/`alice.`/`bob.`/`charlie.` prefixes on `example.com`) — cleanup is attempted after every verification pass and has failed every single time so far with transient `ETIMEDOUT`/`fetch failed` errors against the Supabase admin API from this machine (not a code issue). Harmless (no real data), but this machine's connectivity to the admin API specifically seems unreliable enough that it's worth just doing a bulk cleanup from the dashboard directly at some point rather than continuing to retry per-stage.

## What NOT to Do

- Don't import code directly from `mutiso-main` — it's reference/spec only. Re-derive the pattern, don't copy-paste the implementation (the old code has the bugs and legacy cruft documented in its own `CLAUDE.md`).
- Don't create a table on the remote Supabase project without writing the migration file first — this workspace's entire premise is not repeating the old repo's schema drift.
- Don't use `useState` + manual validation for new forms — use react-hook-form + zod, per the UI standards above.
- Don't use a bare spinner for loading states — use a skeleton.
- Don't match "is this the same worker/item" by comparing name strings — use a foreign key.
- Don't bump Vite past 5.x without first checking `@vitejs/plugin-react-swc`'s peer dependency range.
- Don't re-add the Webshare proxy to `.npmrc` or global git config without checking with the user first — it was removed deliberately after confirming it was blocking legitimate npm/git operations and wasn't needed for this project.
- Don't inline a cross-table subquery directly in an RLS policy if the other table's policy might query back — use a `SECURITY DEFINER` function to break the cycle (see the RLS recursion lesson above). Test any new pair of cross-referencing policies with a real second-user query, not just a mental read-through.
- Don't add direct client-side INSERT/UPDATE policies on `invites`/`user_roles`/`site_assignments` to support the invite-join flow — extend the `consume_invite()` RPC instead. Narrow-looking table policies for "just this one user, just this one row" have already proven to leak cross-tenant in this exact flow.
- Don't trust `!error` as proof an UPDATE did anything under RLS — check the actual returned row count.
- Don't add a `useMutation` that calls `submitOrQueue` without `networkMode: 'always'` — without it, the mutation silently never fires while offline (see Offline-first architecture above).
- Don't call `useOfflineQueue()` outside the `OfflineQueueProvider` tree, and don't create a second `OfflineQueueProvider` instance — multiple independent queue processors is exactly the bug that caused a duplicate-insert 409 during Stage 2 testing.
- Don't queue photo/file uploads through the offline queue — it's designed for small JSON payloads (table inserts / RPC calls), not multi-megabyte files sitting in IndexedDB.
- Don't add owner-only decision actions (payroll generate/mark-paid, variation-order/permit approve-reject) to the offline queue — they'll always be rejected for a non-owner on replay, and every authorized user of an owner-gated action is, by definition, not the field-offline foreman. See "Financial authority ≠ field capture" above.
- Don't `CREATE OR REPLACE` an RPC after adding new parameters — `DROP FUNCTION IF EXISTS` first, or you'll end up with two overloads and the old signature still callable.
- Don't leave an optional `type="date"` input's value going straight into an insert/update — coerce falsy to `null` in the mutation hook, or a blank (very common) submission 400s against Postgres's date parser.
- Don't write a new SECURITY DEFINER RPC that takes a row id and skips its own `owns_site()`/`is_assigned_foreman()` check on the assumption "RLS already covers it" — SECURITY DEFINER bypasses RLS internally by design, so the function itself is the only thing standing between a valid-looking row id and cross-tenant access. See `checkout_tool()`/`return_tool()` for the pattern.
- Don't offline-queue an UPDATE or RPC that targets a specific row's id if that id could itself belong to a not-yet-synced queued insert — see "Referential availability" above for why visitor sign-out isn't queued even though sign-in is.
- Don't reach for a `notifications`-table trigger or `pg_cron` for a simple "expiring within N days" badge — compute it client-side (`isExpiringSoon()`) like insurance/certification expiry already does, and only add scheduled jobs when there's a real push/email requirement to justify the added infrastructure.
- Don't create a new "report an issue" table without first checking whether `incident_log`'s open-ended `category` column already fits — Stage 7's dust/noise/spill reporting reused it entirely rather than duplicating the table + notification trigger.
